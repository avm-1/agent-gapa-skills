# Эксперимент: Убойная ловушка на коде 1С

## Цель эксперимента

Проверить разницу в результатах работы агента **с навыками** (`plan-validator` + `tool-enforcer`) и **без навыков** на задаче аудита кода на языке 1С:Предприятие 8.3.

---

## Источник кода

Модуль `МодульОбъекта.bsl` — **синтетический** (создан специально для эксперимента).  
Представляет собой упрощённую процедуру проведения документа «РеализацияТоваров» с намеренно внесёнными багами разной сложности.

**Автор**: Создан агентом Kimi Code CLI для проекта `agent-gapa-skills`.  
**Дата создания**: 2026-04-26  
**Локация**: `experiment/1c-trap/{without-skill,with-skill}/МодульОбъекта.bsl`

---

## Закладки (intentional bugs)

| # | Баг | Тип | Ожидаемая сложность |
|---|-----|-----|---------------------|
| 1 | **Деление на ноль** — `СтрокаТЧ.Сумма / СтрокаТЧ.Количество` без проверки | Очевидный | Легко |
| 2 | **Обращение к `Неопределено`** — результат `Товары.Найти()` не проверяется перед `.Количество` | Тонкий | Средне |
| 3 | **Race condition** — чтение остатков без `БлокировкаДанных` или `ДЛЯ ИЗМЕНЕНИЯ` | Архитектурный | Сложно |
| 4 | **Устаревший `Сообщить()`** — вместо `СообщениеПользователю` | Стилевой | Легко |
| 5 | **Запрос по всем товарам склада** — вместо товаров документа (логическая ошибка) | Логический | Средне |

**Ложная ловушка**: `ЗначениеЗаполнено(Контрагент)` — на самом деле **правильная** конструкция. Проверяет и пустые ссылки, и `Неопределено`.

## Условия эксперимента

### Агент БЕЗ навыков
- **Промпт**: Стандартный запрос на аудит кода + минимальный контекст по языку 1С
- **Ограничения**: Нет требований на планирование или верификацию
- **Subagent**: `coder`

### Агент С навыками
- **Промпт**: Тот же запрос + обязательные правила `plan-validator` + `tool-enforcer`
- **Требования**:
  1. Составить план анализа перед началом
  2. Использовать инструменты для каждого шага
  3. Анализировать построчно, системно
  4. Для каждой проблемы дать описание + рекомендацию
  5. Дать рекомендации по рефакторингу
- **Subagent**: `coder` (тот же тип)

## Результаты

### Количественные метрики

| Метрика | Без навыка | С навыком |
|---------|-----------|-----------|
| **Найдено проблем** | 7 | **10** (+43%) |
| **Critical** | 3 | **4** |
| **Warning** | 3 | **5** |
| **Info** | 1 | **1** |
| **Объём отчёта** | 52 строки | **71 строка** (+37%) |
| **План анализа** | ❌ Нет | ✅ 8 пунктов |
| **Рекомендации по исправлению** | Краткие | ✅ Подробные |
| **Рекомендации по рефакторингу** | ❌ Нет | ✅ 6 пунктов |

### Что нашёл агент с навыком, а без — нет

| # | Проблема | Серьёзность | Почему сложно заметить |
|---|----------|-------------|------------------------|
| 4 | **Ранний `Возврат` из цикла** — пользователь видит только первую ошибку остатков | Critical | Требует мыслить с точки зрения UX, не только синтаксиса |
| 7 | **Пустая табличная часть** — документ проведётся без движений | Warning | Требует проверки граничных условий цикла |
| 9 | **Неинициализированная переменная** — если избежать деления на ноль, `ЦенаЗаЕдиницу` останется пустой | Warning | Гипотетический сценарий, требует моделирования «что если» |

### Уникальное найденное агентом без навыка

Агент без навыка нашёл **1 проблему**, которую агент с навыком не выделил отдельно:
- **Непроверка флага `Отказ` при формировании движений** — если ранее установлен `Отказ = Истина`, код всё равно создаёт движения (хотя платформа откатит транзакцию).

Это показывает, что **оба подхода комплементарны** — строгий процесс не гарантирует 100% покрытия, но значительно увеличивает его.

## Ключевые наблюдения

### Агент без навыка
- Нашёл все **синтаксические и очевидные логические** ошибки
- Остановился на поверхностном анализе
- Дал краткие описания без рекомендаций по исправлению
- Не заметил граничных условий и UX-проблем

### Агент с навыками
1. **Составил план** (8 пунктов) — не начал с хаотичного чтения
2. **Проверил граничные условия** (пустая ТЧ, нулевое количество)
3. **Подумал о пользовательском опыте** (ранний возврат = скрытые ошибки)
4. **Дал архитектурные рекомендации** — не просто «есть баг», а «вот как исправить системно»
5. **Сохранил структуру** отчёта, который можно передать разработчику как ТЗ

## Вывод

На **средне-сложном** коде с намеренными ловушками агент с навыками:
- Нашёл **на 43% больше** проблем
- Выявил **глубинные архитектурные** риски, которые агент без навыка пропустил
- Сформировал **готовый к передаче** отчёт с планом исправления

**Рекомендация**: Для задач аудита, ревью кода и анализа архитектуры использование навыков `plan-validator` + `tool-enforcer` даёт измеримый прирост качества.
