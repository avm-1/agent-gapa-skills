--- КРИТИЧЕСКИЕ БАГИ (Ошибки времени выполнения / Логические) ---

[БАГ-1] Обращение к свойству Неопределено (Runtime Error)
  Строка: 29-30
  Код:
      СтрокаТовара = Товары.Найти(Результат.Номенклатура, "Номенклатура");
      Если Результат.КоличествоОстаток < СтрокаТовара.Количество Тогда
  Проблема: Метод Товары.Найти() возвращает Неопределено, если строка с указанной 
            номенклатурой отсутствует в табличной части. Обращение 
            СтрокаТовара.Количество вызовет ошибку "Обращение к свойству объекта 
            на значении, не являющемся объектом".
  Рекомендация: Добавить проверку:
      Если СтрокаТовара = Неопределено Тогда
          Продолжить; // или Отказ = Истина с сообщением
      КонецЕсли;

[БАГ-2] Пропуск проверки для товаров с нулевым остатком (Логическая ошибка)
  Строка: 16-35
  Проблема: Запрос читает остатки из регистра. Если товар есть в ТЧ документа, 
            но его остаток на складе = 0, строка в результате запроса не 
            сформируется (Остатки() не возвращает нулевые строки). Следовательно, 
            цикл Пока Результат.Следующий() пропустит этот товар, проверка 
            не выполнится, и документ спишет несуществующий товар.
  Рекомендация: Переделать запрос на ВНУТРЕННЕЕ/ЛЕВОЕ СОЕДИНЕНИЕ табличной части 
            документа с остатками регистра. Таким образом все строки ТЧ 
            попадут в проверку.

[БАГ-3] Деление на ноль
  Строка: 46
  Код:
      ЦенаЗаЕдиницу = СтрокаТЧ.Сумма / СтрокаТЧ.Количество;
  Проблема: Если Количество = 0, произойдет деление на ноль и аварийное 
            завершение процедуры проведения.
  Рекомендация: Добавить проверку:
      Если СтрокаТЧ.Количество <> 0 Тогда
          ЦенаЗаЕдиницу = СтрокаТЧ.Сумма / СтрокаТЧ.Количество;
      Иначе
          ЦенаЗаЕдиницу = 0;
          // Или Отказ = Истина;
      КонецЕсли;

[БАГ-4] Race Condition (конкурентный доступ) — отрицательные остатки
  Строка: 13-35, 56
  Проблема: Остатки читаются в начале процедуры, а движения записываются в конце. 
            Между этими двумя моментами другой пользователь/сеанс может списать 
            тот же товар. Проверка окажется устаревшей, и документ создаст 
            отрицательный остаток на регистре.
  Рекомендация: 
      1. Установить блокировку данных (БлокировкаДанных) ПЕРЕД выполнением 
         запроса остатков по списываемым товарам и складу.
      2. Использовать контроль остатков при записи движений:
         Движения.ТоварыНаСкладах.Записать(Истина); // с контролем
         или передать нужные параметры в зависимости от версии платформы.

--- СЕРЬЕЗНЫЕ ПРОБЛЕМЫ (Устаревшие практики / Надёжность) ---

[ПРОБ-1] Использование устаревшего метода Сообщить
  Строки: 8, 31, 50
  Проблема: В 1С:Предприятие 8.3 метод Сообщить() считается устаревшим. 
            Сообщения не привязаны к полям формы, их сложно обрабатывать 
            программно, они могут не отображаться корректно в веб-клиенте.
  Рекомендация: Заменить на объект СообщениеПользователю с указанием 
            ПутьКДанным (например, "Объект.Контрагент", "Объект.Товары[0].Количество").

[ПРОБ-2] Немедленный Возврат при первой ошибке
  Строки: 10, 33
  Проблема: При обнаружении ошибки процедура сразу выходит. Если в документе 
            несколько ошибок (несколько товаров с недостатком, не заполнен 
            контрагент и склад), пользователь увидит только первую и будет 
            исправлять по одной за раз.
  Рекомендация: Убрать Возврат изнутри циклов/проверок. В конце процедуры 
            добавить:
      Если Отказ Тогда
          Возврат;
      КонецЕсли;

[ПРОБ-3] Отсутствие проверки заполнения Склада
  Строка: отсутствует
  Проблема: В процедуре проверяется только Контрагент. Склад — ключевое 
            измерение для списания остатков — не проверяется на заполненность.
  Рекомендация: Добавить аналогичную проверку:
      Если НЕ ЗначениеЗаполнено(Склад) Тогда
          // СообщениеПользователю
          Отказ = Истина;
      КонецЕсли;

[ПРОБ-4] Дубли строк в табличной части "Товары"
  Строка: 29
  Проблема: Если в ТЧ две строки с одной и той же Номенклатурой, метод Найти 
            вернет только первую. Проверка остатков сравнит суммарный спрос 
            (который может быть больше, чем в первой строке) только с первой 
            строкой. Итоговая сумма по документу может превысить остаток, 
            но это не будет обнаружено.
  Рекомендация: Перед проверкой остатков свернуть табличную часть документа 
            по Номенклатуре с суммированием Количества (временная таблица 
            в запросе или программно).

[ПРОБ-5] Нет проверки корректности числовых полей в ТЧ
  Строки: 38-52
  Проблема: Не проверяется, что Количество > 0 и Сумма >= 0. Отрицательные 
            или нулевые значения могут привести к некорректным движениям 
            или делению на ноль.
  Рекомендация: Добавить валидацию полей ТЧ перед формированием движений.

[ПРОБ-6] Нет обработки исключений
  Строка: вся процедура
  Проблема: Любая непредвиденная ошибка (например, ошибка СУБД при записи 
            движений) прервёт выполнение процедуры и оставит документ 
            в неопределенном состоянии. Хотя платформа управляет транзакцией 
            проведения, отсутствие локальной обработки затрудняет диагностику.
  Рекомендация: Для сложной логики рекомендуется использовать 
            Попытка / Исключение с записью в журнал регистрации.

--- РЕКОМЕНДАЦИИ ПО РЕФАКТОРИНГУ МОДУЛЯ ---

1. СТРУКТУРА:
   - Выделить проверки заполнения шапки и табличной части в отдельную 
     процедуру (например, ПроверитьЗаполнениеДокумента).
   - Выделить формирование движений в отдельную процедуру.

2. ПРОВЕРКА ОСТАТКОВ:
   - Реализовать единым запросом: временная таблица ТЧ документа 
     -> соединение с остатками регистра.
   - Установить БлокировкаДанных.Элементы по Номенклатуре и Складу 
     из ТЧ документа ДО выполнения запроса.

3. СООБЩЕНИЯ:
   - Полностью отказаться от Сообщить().
   - Все сообщения формировать через СообщениеПользователю с привязкой 
     к конкретному реквизиту/строке.

4. ОБРАБОТКА ОШИБОК:
   - Собирать все ошибки (Отказ = Истина), не прерывая процедуру досрочно.
   - Проводить запись движений только если Отказ = Ложь.

5. БЕЗОПАСНОСТЬ ОПЕРАЦИЙ:
   - Добавить проверку деления на ноль.
   - Проверять дубли номенклатуры в ТЧ.
   - Проверять положительность Количества и неотрицательность Суммы.
